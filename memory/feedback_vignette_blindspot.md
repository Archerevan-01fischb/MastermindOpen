# Vignettes vs drops: evaluate structure, not just topic

Rule: When classifying memory files for inclusion in templates, public docs, or sharing — do NOT drop a file just because the CONTENT is hyperspecific to one user. Check whether the STRUCTURE (sections, fields, decision rules, state-tracking shape) is generalizable to other users with similar interests. If yes, promote to a vignette template with placeholders.

**Why:** The meta-error is conflating **specific topic** with **narrow pattern**. A Civ5-strategy memory file is content-specific to one game, but the STRUCTURE (survival rules + per-faction playbook + session log + pain-point diagnosis) generalizes to every strategy game. Same for pond → aquaculture, woodworking → maker projects, home_assistant → smart-home orchestration. A first-pass drop list almost always over-drops because the classifier reads CONTENT and misses STRUCTURE.

Secondary error: productivity bias. Work/health/finance domains feel more "template-worthy" than gaming/hobbies/crafts. For a personal mastermind, the gaming-companion vignette is exactly the kind of fun, recurring relationship that demonstrates value better than another spreadsheet template.

**How to apply:**
- When tempted to drop something as "hyperspecific" or "too narrow": pause and check the STRUCTURE. Does it have sections / fields / rules that a different user with a different domain instance could fill in? If yes → vignette candidate, not a drop.
- Don't accept "this is just one user's hobby" as a reason to drop. Hobbies are exactly where templates show their value to readers.
- When asking an Explore agent to classify files for sharing: include explicit "look for transferable patterns even in specific content" framing. The first agent's classifications can be wrong; a second-pass critique pays off.
- For sharing / publishing decisions specifically: include at least one vignette in the hook because abstract scaffolding pitches convert worse than concrete uses.
